## 4. このPMの前提
- レイアウトは既に確定事項として扱う
- 見た目はできる限り変えない
- 目的はデザイン変更ではなく、整理、保守性向上、責務分離、バックエンド接続前の土台整理
- `index.html` を基準ページとして扱う箇所がある
  - 特にヘッダー、共通導線、共通ラベルの見え方
- バックエンドは未接続で、現状はフロントモック段階
- 予約完了後の正式導線や自動遷移の扱いは、バックエンド実装後に再検討事項が残っている

## 5. 作業前確認ルール
- 作業前に必ず次の3点を明示する
- 1. 対象ファイル
- 2. 変更意図
- 3. HTMLを触るのか、CSSだけなのか、Markdownだけなのか

### 5-1. 禁止事項
- ユーザー確認なしに大きな構造変更をしない
- 無関係なラッパーや `div` を追加しない
- 共通化できそうという推測だけで `common.css` に広く効く変更を入れない
- ブラウザ未確認の内容を「揃った」「完了した」と断定しない
- 依頼された確認より先に、原因説明や別提案を長く出さない

## 6. HTML / CSS の運用方針
### 6-1. 基本
- セマンティックなHTMLを維持する
- 共通責務は `css/common.css` に寄せる
- ページ固有責務はページ別CSSに残す
- 共通化の条件は「複数ページで同一の責務を持つこと」

### 6-2. 共通CSSに入れてよいもの
- `site-header`
- `page-header-inner`
- `brand`
- `page-main`
- `page-body`
- `page-panel`
- `page-section`
- `page-section-inner`
- `btn`
- `btn-primary`
- `btn-secondary`
- `eyebrow`
- `section-label`
- `glass-panel`
- `status-pill`

### 6-3. ページ別CSSに残すもの
- ページ固有の `main-nav`
- ページ固有のCTAサイズや配置
- 質問ページのステップUI
- 予約ページ固有のフォームレイアウト
- グラフページ固有のグラフ / スライダー構成
- 完了ページ固有の詳細カードや案内構成

### 6-4. ヘッダー運用
- ヘッダーの基準は `index.html`
- まず `site-header` / `page-header-inner` / `brand` に当たっているCSSを確認する
- 見た目差がある場合は、先に以下を確認する
- 1. `--container`
- 2. `site-container`
- 3. `page-header-inner`
- 4. `brand`
- 5. ページ固有の `main-nav` / `header-cta`

## 7. Markdown運用ルール
### 7-1. チェック表記
- `[〇]` 完了
- `[△]` 進行中
- `[ ]` 未着手
- `[未]` 未実装
- `[保]` 保留

### 7-2. 記述ルール
- 実際に反映済みの内容だけを完了にする
- 未確認の表示確認は未確認と書く
- ユーザー確認待ちのものは完了にしない
- 将来対応やバックエンド依存は `[保]` か `[未]` で分ける
- 「未着手」と「未実装」を混同しない
  - 未着手: 作業対象だがまだ手を付けていない
  - 未実装: 方針や必要性はあるが、現時点でコードや仕組み自体がない

## 8. 今回起きたインシデントと再発防止
### 8-1. 起きたこと
- ヘッダー差分の原因確認より先に、局所的な共通化を進めた
- `questionnaire.html` / `questionnaire_step2.html` に不要な構造追加を行った
- Material Icons に影響する変更を入れてしまった
- ユーザーが見ているPC基準ではなく、先にモバイル方向で判断してしまった
- 実態確認が不足したまま、見た目が揃った前提で進めてしまった

### 8-2. 原因
- 設計資料、引継ぎMarkdown、全体CSSを見切る前に局所判断した
- `brand` 単体だけを見て、実際には `container` と親レイアウトの影響が大きいことを後追い確認した
- `.eyebrow` の見え方も、要素自体ではなく親の `grid` 構造差が原因だった

### 8-3. 再発防止
- 変更前に「どのCSSがどこへ当たっているか」を一覧で確認する
- 問題が見た目でも、最初に PC / 中間 / SP のどの話かを切り分ける
- 原因確認前にHTML構造を変えない
- 差し戻しが入った場合は、まず誤変更の撤回を優先する
- 原因説明は、依頼された確認や修正の後に必要最小限で行う

## 9. インシデント時の対応手順
- 1. 直前変更ファイルを特定する
- 2. 影響範囲をHTML / CSS / JS / Markdownに分ける
- 3. 構造変更が入っている場合は最優先で確認する
- 4. 必要なら誤変更を戻す
- 5. 戻した内容と未解決内容を分けて共有する
- 6. 再調査の対象ファイルと観点を明示してから次の修正に入る

## 10. 作業終了時のルール
- 何を変更したかをファイル単位で残す
- 何を確認済みで、何が未確認かを書く
- 次にやるべきことを、完了済みと混ぜずに分けて書く
- 必要なら `作業まとめ` と `直近チェックリスト` の両方を更新する

## 11. 次回以降の標準フロー
- 1. 設計資料と最新引継ぎMarkdownを読む
- 2. 対象ファイルと変更意図を事前共有する
- 3. 関連HTML / CSS / Markdownを全体確認する
- 4. 最小差分で修正する
- 5. 必要なMarkdownだけ更新する
- 6. 未確認事項を未確認のまま正確に残す
